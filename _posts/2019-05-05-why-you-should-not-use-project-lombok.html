Project lombok comes with many use full features to reduce boiler plate code. But it comes with the a cost.
Lets go through how java compilation process before going to internals of Lombok.

Java compilation process.

As per the documentation https://openjdk.java.net/groups/compiler/doc/compilation-overview/index.html

To make it short, following steps happen during .java to .class file
  1. Java file enters Parse and Enter phase, where .java files are converted to AST (Abstract Systax Tree)
  2. Annotation Processing - If any annotation processing generate any new source file, then .java source is created and compilation is restarted with step 1.
  3. Analyse and generate - AST's created are parsed and conveted to .class files 
Please not at any of the above stages .class files are NOT modified, java specification DOES NOT allow to modify the class files
Then how does Project Lombok edits the class file to generate getters/setters, toString, etc.

Lombok internall uses  source from
com.sun.tools.javac.tree.* packages, which is NOT a part of supported API.

Java doc clearly says as follows,

This is NOT part of any supported API.
If you write code that depends on this, you do so at your own risk.
This code and its internal interfaces are subject to change or
deletion without notice.
Reference: Javadoc from JCTree

Project lombok's implementation sits on set of unspported API's, which may be changed/deleted anytime,
which is a huge risk for up coming JAVA releases.

Stay away from libraries which directly modifies Abstract syntax Trees like
lombok, powermock, etc.
