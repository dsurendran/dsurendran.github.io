After conversing with lot of people on Dependency Inject and IOC, found even a lot of experienced folks miss the concept, so I decided to write this blog.

What is Dependency Injection?

As per Wikipedia its bit confusing: Dependency injection is a software design pattern that implements inversion of control for resolving dependencies… (stop stop…???? What is implenting inversion of control ???)

IOC Is basically we are inverting the controls of our program to a framework to manage the flow of control, life cycle of objects etc(some IOC containers are Spring, Google Guice, etc, these frameworks also provides Dependency Injection)

Then what is DI?

Wikipedia Overview is bit more close to this:
Dependency injection separates the creation of a client's dependencies from the client's behavior, which allows program designs to be loosely coupled.

While using DI our class does not need to be aware of the Service implementation until the run time, this is very helpful in following cases

-          Reducing the boilerplate code.
-          Easier unit testing by mocking the services

Let's take the same Martin Fowler’s example. (Assume we have only one implementation of MovieFinder)

Below is the code without DI


public class MovieLister{
  private MovieFinder movieFinder;
  public MovieLister(){
    movieFinder = new MovieFinder();
  } 
  public List findAll(){
      return movieFinder.getAllMovies();
  }
}


In the above code in MovieLister(), we are creating new instance of MovieFinder object,

1.       our code dependent on MovieFinder() object
2.       we cannot mock MovieFinder() in this case.

How do we overcome this by dependency injection.

There are different ways we can provide DI as follows
       1.       Setter injection

public void setMovieFinder(final MovieFinder mf){
  this.movieFinder = mf;
}

2.       Constructor injection

Public MovieLister(final MovieFinder mf){
  this.movieFinder = mf;
}

3.       Interface injection (I think this is obsolete and useless till we have more than 1 implementation of an interface)

1 & 2 above removes the new MovieFinder() out of MovieLister, so that dependency of the code is removed out of the MovieLister and the code can be unit tested by mocking and setting the MovieFinder object to MovieLister.

1.       So Do we need a separate framework like Spring only for Dependency Injection?
Simple answer is big NO.
Checkout this http://programmers.stackexchange.com/questions/300127/why-do-we-need-frameworks-for-dependency-injection
2.       I don’t have any unit test cases and there is only one implementation of all my services. Should I use Dependency Injection?
NO you don't need DI ( and all the best maintaining the code).

Use spring or any other frameworks ONLY when you want to reduce boilerplate code for creation of objects(services) or if you need other functionalities like managing Transactions, AOP, handling exceptions, etc
